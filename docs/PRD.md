# PRD — RS 파이프라인 신뢰성 복구 및 전체 재구축

## Problem Statement

PM의 목표는 "미국 주식 약 4,600개가 **매 거래일 정확하고 신뢰할 수 있는 RS Rating으로 저장**되는 시스템"이다. 이 목표는 2026-04-17을 마지막으로 깨졌다.

- 2026-04-20부터 종가가 있는 종목이 4,593개 → 75개로 절벽처럼 붕괴했고, 5/20부터 54개만 남았다.
- 4,636종목 중 4,518개(97%)가 2026-04-17에 동결됐다. 남은 종목이 "알파벳 앞쪽(A~ACNT) + SPY/QQQ"라는 점이 원인 버그의 데이터 지문이다.
- 2026-05-25부터는 yfinance 라이브러리 breaking change로 GitHub Actions가 매일 완전히 실패했다(실패 이메일은 정상 발송되어 PM이 인지).
- 더 위험했던 4/17~5/22 구간은 워크플로우가 "성공"으로 표시되어 실패 이메일조차 오지 않은 "조용한 정체"였다.

원인은 단일 버그가 아니라 맞물린 결함들이다(상세: `docs/DECISIONS.md`, 진단 근거: 코드 + Supabase 실데이터 교차검증).

## Solution

근본 결함을 모두 수정하고, 신뢰 구간 전체를 깨끗이 재생성하며, 재발을 구조적으로 차단한다.

1. **다운로드 정체 차단** — 글로벌 최신-날짜 커서를 고정 트레일링 윈도우로 교체.
2. **RS 계산 정확성** — 종목별 유효 거래일 기준 ROC, 모집단/히스토리 정의 확정, 기준 지수 처리 정리.
3. **외부 의존성 봉인** — 정확 버전 고정 + lock 파일, yfinance 실패 감지를 반환 커버리지 검사로 교체.
4. **데이터 재구축** — 2025-03-21~현재 전체 RS 재계산(가격 재활용).
5. **운영 가시성** — "조용한 정체"를 자가 점검으로 탐지해 워크플로우를 실패시켜 기존 이메일 알림이 잡게 함.
6. **회귀 방지** — 깨진 3경로에 결정론적 테스트.

## User Stories

1. 데이터 운영자로서, 매 거래일 모든 충분히-오래된 종목의 종가가 빠짐없이 채워지길 원한다 — 그래야 RS가 전체 유니버스 기준으로 정확하다.
2. 데이터 운영자로서, 어느 한 종목이 일시적으로 다운로드에 실패해도 다음 실행에서 자동으로 메워지길 원한다 — 정체가 영구화되지 않도록.
3. 데이터 운영자로서, 다운로드가 항상 "최근 N거래일"을 모든 종목에 재요청하길 원한다 — 글로벌 커서가 일부 종목만 전진시키는 일이 없도록.
4. 투자자로서, 각 종목의 RS Rating이 "그날 살아있는 소수 종목"이 아니라 "충분한 히스토리를 가진 전체 유니버스" 대비 백분위이길 원한다.
5. 투자자로서, 유효 종목이 임계치(예: 90%) 미만인 불완전한 거래일에는 RS Rating이 저장되지 않길 원한다 — 가짜 등급이 DB에 남지 않도록.
6. 투자자로서, 252거래일 미만 히스토리의 신규 종목은 RS가 없길 원한다 — 짧은 데이터로 만든 비교 불가능한 등급이 섞이지 않도록.
7. 투자자로서, 거래정지·내부 결측이 있는 종목도 ROC가 "진짜 n거래일 전" 종가를 기준으로 계산되길 원한다 — 행 위치로 잘못 계산되지 않도록.
8. 투자자로서, SPY/QQQ가 RS Rating을 받아 "지수가 전체 종목 중 상위 몇 %"인지 확인할 수 있길 원한다(일반 ETF는 유니버스에서 제외).
9. 데이터 운영자로서, yfinance가 일부 종목 데이터를 반환하지 않을 때 그 누락을 정확히 감지·기록하길 원한다 — `yf.shared` 같은 라이브러리 내부에 의존하지 않고.
10. 데이터 운영자로서, 외부 라이브러리(yfinance/pandas/finviz)가 새 버전을 내도 CI가 어제와 같은 버전으로 도는 재현성을 원한다.
11. 라이브러리 사용자로서, PyPI 클라이언트(rs_rating) 설치 시 의존성 충돌이 없길 원한다 — CI의 정확 고정이 배포 패키지를 옭아매지 않도록.
12. 데이터 운영자로서, close 가격은 13개월만 보존되어 매일 계산량이 일정하길 원한다(타임아웃 방지). 단 RS 등급은 영구 보존되어 과거 조회가 가능하길 원한다.
13. PM으로서, 워크플로우가 "성공"으로 표시되지만 실제로는 정체된 상황을 탐지하길 원한다 — 자가 점검이 데이터 완전성을 검사해 비정상이면 워크플로우를 실패(빨강) 처리.
14. PM으로서, 정체가 발생하면 이미 설정된 GitHub 실패 이메일로 통지받길 원한다 — 새 알림 채널 없이.
15. 데이터 운영자로서, 전체 재구축을 로컬에서 수동으로, Supabase 백업을 먼저 뜨고, 소규모(100종목) 검증 후 전체로 안전하게 실행하길 원한다.
16. 데이터 운영자로서, 재구축이 2025-03-21~현재 범위의 RS를 전부 재계산하고 4/20~5/22의 깨진 값을 전량 덮어쓰길 원한다.
17. 개발자로서, `init.yml`이 engine 의존성을 설치하길 원한다 — 재구축/초기화가 ModuleNotFoundError로 죽지 않도록.
18. 개발자로서, "여러 종목이 서로 다른 날짜에 멈춘 상태에서 update를 돌리면 모든 종목이 같은 말일까지 채워지는지"를 검증하는 회귀 테스트를 원한다.
19. 개발자로서, "결측 종목의 RS가 어떻게 처리되는지"와 "yfinance 누락 감지"를 검증하는 회귀 테스트를 원한다 — 모두 네트워크 없이 결정론적으로.
20. PM으로서, 데이터 의미가 바뀐 만큼 CHANGELOG와 버전이 갱신되어 시스템 변화를 추적할 수 있길 원한다.

## Implementation Decisions

도메인 용어는 `CONTEXT.md` 기준. 모듈/인터페이스는 동작 계약 수준으로만 기술(파일 경로·줄 번호는 슬라이스에서 stale).

- **다운로드 (prices)** — `download_update`는 글로벌 `get_latest_price_date()` 대신 "오늘 − N거래일"을 모든 종목 공통 start로 사용(트레일링 윈도우, N=7~10). upsert는 idempotent라 중복 안전. 글로벌-MAX start 로직 제거.
- **RS 원점수 (compute_rs_raw)** — 각 종목 시계열을 유효 거래일만으로 정렬한 뒤 n거래일 전 종가와 비교(달력 행 shift 제거). 252거래일 미만이면 RS 없음.
- **RS 등급 (compute_rs_rating)** — 모집단 = 그날 RS Raw가 유효한 종목 전체. 유효 종목이 전체 유니버스의 임계치(예: 90%) 미만이면 그날 등급을 산출하지 않음. SPY/QQQ는 모집단에 포함. `reference_tickers`의 죽은 파라미터·거짓 주석 정리.
- **RS 저장 (calculate_and_store)** — 재구축은 recalc_all 전량 재계산. 글로벌 `get_latest_rs_date` 증분 커서가 뒤처진 종목의 백필을 막지 않도록 함(증분 경로도 종목별 결측을 메우게 하거나 재구축 시 전량 경로 사용).
- **실패 감지 (prices)** — `getattr(yf.shared, "_ERRORS")` 제거. 요청 티커 집합 − 반환 티커 집합 = 실패로 기록. 누락분은 선택적으로 개별 재확인.
- **Retention (db)** — close를 13개월 경과분 prune하는 연산 신규. rs_rating은 보존.
- **의존성 고정** — CI 설치용 `requirements.lock`(정확 버전). pyproject의 배포 의존성은 느슨 유지. `daily_update.yml`/`init.yml`이 lock을 사용.
- **운영 가시성** — `status`(또는 전용 점검)가 "최신 거래일에 종가/등급이 빠진 종목 수"를 산출. 임계치 초과 시 비정상 종료(exit 1)해 워크플로우 실패 → 기존 이메일 발송.
- **워크플로우 (init.yml)** — engine extras(또는 lock) 설치하도록 수정.

## Testing Decisions

좋은 테스트 = 공개 인터페이스로 관찰 가능한 행동 검증, 네트워크 없이 결정론적, 리팩터에도 생존. yfinance 다운로드는 경계에서만 mock.

- **오케스트레이션 회귀(주)** — 메모리 SQLite에 종목 A는 4/17까지, B는 5/22까지 넣고 update 실행 → A가 4/18~5/22 날짜에도 채워지는지 검증(정체 근본 수정의 RED 테스트). yfinance 경계 mock.
- **엔진 회귀(보조)** — 결측 종목이 섞인 가격 행렬에서 ROC가 종목별 유효일 기준으로 계산되는지, 모집단 임계치 미만 날 등급이 비는지, 252일 미만 종목이 RS 없이 처리되는지.
- **실패 감지** — 일부 티커만 반환하는 mock 다운로드에서 누락 티커가 정확히 실패로 기록되는지.
- 기존 테스트 50개(test_client/db/rs/splits)는 회귀 보호로 유지. 깨진 경로에만 테스트 신설.

## Out of Scope

- **finviz 대체 마이그레이션** — unmaintained 스크레이퍼지만 현재 작동하며 월 1회만 호출. 별도 조사 슬라이스로 분리(이번엔 버전 고정 + 빈 목록 방어만).
- **RS 공식 가중치 변경** — 0.4/0.2/0.2/0.2 유지(과거 연속성).
- **forward-fill** — 죽은 가격으로 만든 가짜 모멘텀이라 배제.
- **새 알림 채널(Discord/Slack)** — 기존 이메일 + 정체 자가점검으로 충분.
- **UI/차트** — 데이터 파이프라인 범위 밖.
- **2024년(워밍업 이전) RS 확보** — 가격 백필이 부족해 의미 적음.
